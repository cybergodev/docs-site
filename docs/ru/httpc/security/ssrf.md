# Защита от SSRF

SSRF (Server-Side Request Forgery, подделка серверных запросов) — это атака, при которой злоумышленник использует сервер для отправки запросов во внутреннюю сеть. HTTPC включает защиту от SSRF по умолчанию.

## Поведение по умолчанию

```go
cfg := httpc.DefaultConfig()
// AllowPrivateIPs = false → по умолчанию блокируются все приватные IP
```

Заблокированные по умолчанию диапазоны IP:

| Диапазон | CIDR | Описание |
|----------|------|----------|
| IPv4 loopback | `127.0.0.0/8` | localhost |
| Частные класса A | `10.0.0.0/8` | Внутренняя сеть |
| Частные класса B | `172.16.0.0/12` | Внутренняя сеть |
| Частные класса C | `192.168.0.0/16` | Внутренняя сеть |
| Линковый локальный | `169.254.0.0/16` | Автоматическая конфигурация |
| IPv6 loopback | `::1/128` | localhost |
| IPv6 локальный | `fc00::/7` | Уникальные локальные адреса |
| IPv6 линковый | `fe80::/10` | Линковые локальные |

## Освобождение CIDR

В микросервисной среде может потребоваться доступ к внутренним сервисам:

```go
cfg := httpc.DefaultConfig()
cfg.Security.SSRFExemptCIDRs = []string{
    "10.0.0.0/8",       // Внутренний VPC
    "100.64.0.0/10",    // Tailscale VPN
    "172.20.0.0/16",    // Kubernetes Service CIDR
}
```

:::warning Предупреждение
Освобождённые CIDR должны быть максимально точными. Избегайте использования слишком широких диапазонов (например, `0.0.0.0/0`), так как это равносильно отключению защиты от SSRF.
:::

## Защита от DNS Rebinding

HTTPC использует паттерн «разрешение-проверка-прямое подключение» для предотвращения атак DNS Rebinding:

1. Разрешение доменного имени в IP-адрес
2. Проверка всех полученных IP на принадлежность к приватным адресам
3. Прямое подключение к проверенному IP (без повторного разрешения домена)

```go
// Сценарий атаки:
// 1. Злоумышленник контролирует DNS для evil.com
// 2. Первое разрешение возвращает публичный IP (проходит проверку)
// 3. При реальном подключении DNS возвращает 127.0.0.1 (обход проверки)
//
// Защита HTTPC: после проверки используется напрямую проверенный IP без повторного разрешения
```

## Проверка SSRF при редиректах

Цели редиректов также проходят SSRF-проверку:

```go
// Предположим, запрос к public-api.com, сервер возвращает 302 редирект на http://169.254.169.254/
// HTTPC проверит IP цели редиректа и заблокирует доступ к сервису метаданных
```

### Белый список доменов редиректов

```go
cfg := httpc.DefaultConfig()
cfg.Security.RedirectWhitelist = []string{
    "api.example.com",
    "auth.example.com",
    "*.cdn.example.com",  // Поддерживаются подстановочные символы
}

// Редиректы на домены, не входящие в белый список, будут заблокированы
```

## Защита метаданных облачных сред

Адреса сервисов метаданных на облачных платформах:

| Платформа | Адрес | Описание |
|-----------|-------|----------|
| AWS | `169.254.169.254` | Метаданные инстанса |
| GCP | `metadata.google.internal` | Сервис метаданных |
| Azure | `169.254.169.254` | Метаданные инстанса |
| Alibaba Cloud | `100.100.100.200` | Сервис метаданных |

HTTPC по умолчанию блокирует доступ к метаданным AWS/Azure (`169.254.169.254` находится в списке блокировки `169.254.0.0/16`). Метаданные GCP (`metadata.google.internal`) блокируются через проверку DNS-разрешения.

:::warning Предупреждение
Метаданные Alibaba Cloud (`100.100.100.200`) находятся в диапазоне CGNAT (`100.64.0.0/10`), который **не** включён в список блокировки по умолчанию для поддержки VPN, таких как Tailscale/WireGuard. Если требуется защита от доступа к метаданным Alibaba Cloud, используйте дополнительные политики безопасности (например, правила брандмауэра).
:::

## Полное отключение защиты от SSRF

Используйте только в тестовой среде:

```go
// TestingConfig отключает защиту от SSRF
client, _ := httpc.New(httpc.TestingConfig())

// Или ручная конфигурация
cfg := httpc.DefaultConfig()
cfg.Security.AllowPrivateIPs = true
```

:::danger Опасность
В производственной среде никогда не устанавливайте `AllowPrivateIPs = true`.
:::

## Лучшие практики

1. Используйте `SecureConfig()` как базовую конфигурацию безопасности
2. Освобождайте только необходимые диапазоны CIDR
3. Настройте `RedirectWhitelist` для ограничения целей редиректов
4. Регулярно проверяйте конфигурацию `SSRFExemptCIDRs`
5. Используйте промежуточное ПО аудита для логирования всех запросов
